# subagent-collaboration.md

## 文档定位

本文档规定复杂任务如何使用 subagent / 多 Agent 协作，以保持主 Agent 上下文清洁，并避免复杂任务不拆分或简单任务过度拆分。

## 启动前决策

每个开发或机制修改任务开始前，主 Agent 必须判断是否使用 subagent，并在计划中写明：

```text
subagent：使用 / 不使用
理由：任务复杂度、风险、验证需求
```

## 必须优先使用 subagent 的场景

满足任一条件时，必须优先使用 subagent：

```text
1. 同时涉及前端、后端、数据库、权限中的两类及以上。
2. 涉及多个页面、复杂表单、子表、抽屉、流程流转或原型复刻。
3. 涉及视觉还原、截图对比、复杂交互顺滑性检查。
4. 涉及高风险 bug、原因不明的构建失败、接口联调失败或权限异常。
5. 单次修复失败后需要第二轮定位。
6. 需要开发与 QA 独立分工验证。
7. 用户明确要求使用多 Agent / subagent。
```

## 推荐角色分工

| 角色 | 负责内容 | 不负责内容 |
|---|---|---|
| 主 Agent | 读取规则、拆分任务、控制边界、最终审查 | 长时间陷入全部细节 |
| 开发子 Agent | 最小实现或修复 | 扩大范围、改无关模块 |
| QA 子 Agent | 验证、构建、截图/交互检查 | 擅自改业务逻辑 |
| 文档子 Agent | 更新 Spec、日志、组件文档 | 改代码实现 |

## 子任务交接要求

分配子任务时必须给出：

```text
1. 任务目标；
2. 允许修改范围；
3. 禁止修改范围；
4. 必读文档；
5. 验证方式；
6. 输出格式。
```

子 Agent 返回时必须给出：

```text
1. 修改文件；
2. 修改摘要；
3. 验证结果；
4. 失败或未验证原因；
5. 风险与遗留问题。
```

## 失败回路

```text
QA 输出失败证据
→ 主 Agent 判断是否仍在原任务范围内
→ 开发子 Agent 最小修复
→ QA 子 Agent 复测
→ 通过后主 Agent 最终审查
```

同一问题连续失败 3 次时，应停止盲修，考虑局部重构或重新收窄任务。
